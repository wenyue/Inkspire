# 作品命名机制设计

## 背景

当前中国画标题由题材、气质、设色、笔墨和形制稳定映射到通用标题池。名作参考模式不会提交这些普通创作答案，因此常从通用池得到与所选原作无关的名称。现有避重逻辑还会改选标题池中的其他名称，只有候选全部占用后才使用阿拉伯数字形式的 `其2`，不符合新的产品规则。

## 目标

- 为策展目录中的每幅名作人工配置 5 个贴合原作题材、意境和画面特征的新作品名。
- 名作参考、自由生成和书法统一按当前收藏处理重名。
- 重名后缀使用 `其一`、`其二`、`其三` 等中文序数，并复用最小可用序号。
- 只以当前用户收藏中 `favorite != 0` 的作品为重名边界；移出收藏的名称可以重新使用。
- 不修改任何已有作品标题，不改变现有 API 字段和作品记录结构。

## 非目标

- 不调用运行时 AI 为作品题名。
- 不增加英文作品标题或多语言标题字段。
- 不刷新或重新下载名作图片资源。
- 不改变绘画提示词、生成队列、并发策略或图片处理流程。

## 权威数据与生成产物

`scripts/classic-artwork-curation.mjs` 继续作为名作策展数据的权威来源。每条策展记录新增 `newArtworkTitles`，值为恰好 5 个非空、互不重复的中文标题。500 个策展标题在整个目录中保持唯一，避免不同原作天然争用同一个基础标题。

`scripts/build-classic-artworks.mjs` 和 `scripts/apply-classic-artwork-curation.mjs` 将该字段写入生成清单 `config/classic-artworks.json`，对外字段名为 `new_artwork_titles`。不得直接手工维护生成清单中的该字段。

`scripts/validate-classic-artworks.mjs` 和服务端配置测试验证：

- 每幅名作都有 `new_artwork_titles`；
- 数组长度严格为 5；
- 每个标题去除首尾空白后仍非空；
- 同一名作内和整个 100 幅名作目录内均无重复标题。

标题只使用中文，不为 `zh-Hant` 或 `en` 建立并行版本。这与当前生成作品仅保存一个中文标题的模型保持一致。

## 标题选择

服务端 `server/src/jobs.js` 继续拥有标题生成和作品创建时的避重决策。客户端传来的 `classic_artwork_title`、`classic_artwork_category` 或候选标题均不作为权威数据；服务端只根据 `classic_artwork_id` 在已加载的 `config.classicArtworks` 中查找候选名称。

### 名作参考模式

1. 根据 `classic_artwork_id` 取得该名作的 5 个策展标题。
2. 按配置顺序选择第一个尚未作为完整标题出现在当前收藏中的名称。
3. 五个基础名称都已存在时，以第一个名称为基础，添加最小可用中文序数后缀。

例如某名作的五个名称依次为 `溪山清韵`、`云壑松声`、`烟岚归舟`、`松风入画`、`远水含光`：前五次依次使用这五个名称；之后使用 `溪山清韵 其一`、`溪山清韵 其二`。

如果名作配置异常或旧测试配置没有候选字段，则回退到自由生成的基础标题规则，保证旧配置和测试夹具仍可运行。正式策展清单由验证器保证不会触发该回退。

### 自由生成模式

继续根据 `painting_subject`、`painting_mood`、`painting_palette`、`painting_brushwork` 和 `painting_format` 稳定计算一个基础标题。若基础标题已经存在于当前收藏，不再改选标题池中的下一项，而是直接添加最小可用中文序数后缀。

相同创作答案因此会得到稳定的标题系列，例如 `云岫清音`、`云岫清音 其一`、`云岫清音 其二`。

### 书法模式

书法继续以用户提交的完整书写内容作为基础标题。若当前收藏已有完全相同的基础标题，则应用相同的中文序数后缀规则。标题数据保持完整，界面仍只通过现有单行省略样式处理展示宽度。

## 中文序数与碰撞规则

新增的标题碰撞解析逻辑接收一个基础标题和当前收藏标题集合：

1. 基础标题不存在时，直接返回基础标题。
2. 基础标题存在时，从正整数 1 开始查找最小可用序号。
3. 序号转换为常用中文数字，组合为 `<基础标题> 其<中文序数>`。

至少覆盖个位、十位、百位和千位，例如 `其一`、`其十`、`其十一`、`其二十`、`其一百零一`。序号查找不假设标题连续；若收藏中已有原名、`其一` 和 `其三`，则返回 `其二`。

旧数据中的 `其2` 保持原样，不做迁移。新逻辑只生成中文序数格式。

## 数据流

1. 策展源定义每幅名作的 5 个新作名。
2. 构建或策展应用脚本把候选名称写入名作清单。
3. 服务端配置加载完整名作清单。
4. 创建作品时，作业管理器读取当前用户保留中的标题。
5. 作业管理器按创作模式取得基础标题或候选序列，再解析重名。
6. 最终标题同时写入作品记录和作业元数据，保持现有持久化及前端展示契约。

## 测试与验证

实现采用测试先行：

- 服务端作业测试先覆盖自由生成和书法的 `其一`、最小缺号复用，以及名作五个候选依次使用、超过五次后添加后缀。
- API/存储测试确认只检查 `favorite != 0` 的当前收藏，移出收藏后基础标题可再次使用。
- 配置测试和名作验证脚本覆盖 100 幅作品、每幅 5 个标题及全目录唯一性。
- 聚焦测试通过后运行完整服务端测试，并运行 `npm run validate:classic-artworks`。
- 本变更不修改提示词或图片生成管线，因此无需运行真实 Codex 图片生成验证。

## 兼容性与风险控制

- 现有记录不迁移、不重命名。
- HTTP 请求和响应字段不变；新增字段只属于共享名作配置。
- 服务端通过可信 `classic_artwork_id` 读取名称，避免客户端伪造候选列表。
- 500 个标题属于策展内容，机械模板容易产生重复或与原作不贴合；实现时需要逐条审阅，并由验证器阻止遗漏和重复。
